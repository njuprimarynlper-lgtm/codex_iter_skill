# 项目信息补齐向导

当项目刚初始化，或核心文件里仍有空白、`待填写`、`TODO` 时，使用本向导。目标是让 Codex 主动告诉用户需要输入哪些信息、为什么需要，并给出可模仿的例子。

## 工作原则

- 不要只说“请补齐信息”。要列出具体字段、用途、例子和可暂时跳过的项。
- 先问会阻塞下一步的最小信息集，再逐步深入。
- 一次最多问 5 个关键问题；如果信息很多，分批问。
- 用户问“下一步做什么”时，不要只给一个下一步。要先识别当前工作焦点，再给围绕焦点的未来 3 到 5 步路线图、推荐顺序、首选方案和备选方案。
- 用户指令不明确且会影响方向、版本、数据、实验结论或生产配置时，必须先确认；只对低风险实现细节使用暂定假设。
- 用户不知道时，允许写“暂定假设”，并在 `STATUS.md` 标成待确认。
- 用户回答后，把信息写入对应文件：`TASK_SPEC.md`、`data/schema.yaml`、`pipeline/skill_prompt.txt`、`STATUS.md` 或 `PROJECT_INTAKE.md`。

## 主动触发条件

满足任一条件时，Codex 应主动启动信息补齐或下一步诊断，不等用户专门要求：

- 用户说“下一步做什么”“今天做什么”“继续”“帮我看当前状态”“该推进什么”。
- `STATUS.md` 缺少当前工作焦点、当前思考层级、当前阶段、阻塞项或下一步。
- `TASK_SPEC.md` 缺少目标、输入、输出、主指标或验收线。
- `data/schema.yaml` 缺少字段、标签、关系、边界规则或冲突规则。
- `pipeline/skill_prompt.txt` 为空，或只有占位内容。
- `grid.yaml`、`variants/manifest.yaml` 或 `experiments/CHANGELOG.md` 与当前阶段不匹配。

## 下一步诊断输出格式

当用户问“下一步做什么”时，优先输出这个结构：

```markdown
## 当前判断
- 当前工作焦点：
- 当前思考层级：
- 当前项目阶段：
- 主要阻塞：

## 推荐路线图
| 顺序 | 建议动作 | 所属层级/盒子 | 为什么现在做 | 预期产出 |
| --- | --- | --- | --- | --- |
| 1 |  |  |  |  |
| 2 |  |  |  |  |
| 3 |  |  |  |  |

## 候选方案
- 首选方案：
- 备选方案 A：
- 备选方案 B：

## 立即执行

## 需要你补充的信息
1. 

## 我会更新的文件
- 
```

如果信息足够，不要只给建议，要先执行推荐路线图中排序第一的可执行动作，并说明后续几步何时继续。只有当缺少会影响方向判断的信息时，才向用户提问。

## 四类必补信息

### 1. TASK_SPEC.md：任务目标、输入、输出、指标

需要问：

1. 这个项目要解决的业务问题是什么？
2. 输入是什么，最好给 1 到 3 条真实或脱敏样例。
3. 期望输出是什么，最好给一个 JSON、表格或文字示例。
4. 主指标是什么，业务上达到多少算可接受？
5. 有哪些明确不做的事情或边界？

示例答案：

```text
目标：从客服对话中抽取客户投诉原因和责任部门。
输入：一段客服对话文本，包含用户问题和客服回复。
输出：JSON，包含 complaint_reason、department、evidence_span。
主指标：complaint_reason 的 micro-F1，第一阶段验收线 0.82。
非目标：暂不判断责任人，只判断责任部门。
```

### 2. data/schema.yaml：字段、标签、关系、边界规则

需要问：

1. 每条数据有哪些字段？哪些字段必填？
2. 要识别哪些标签、类别、实体或关系？
3. 标签之间是否有层级、互斥或依赖关系？
4. 边界规则是什么？例如实体要不要包含后缀、标点、单位。
5. 遇到模糊样本时按什么规则处理？

示例答案：

```text
字段：id、text、labels。
实体：PERSON、ORG、LOCATION。
关系：WORK_AT(PERSON, ORG)、LOCATED_IN(ORG, LOCATION)。
边界：地点实体包含行政区后缀，如“鼓楼区”；不包含前置介词。
冲突：同一片段可同时是 ORG 和 LOCATION 时，优先按上下文功能判断。
```

### 3. pipeline/skill_prompt.txt：当前生产 Skill 或提示词

需要问：

1. AI 在这个任务里扮演什么角色？
2. 它必须遵守哪些输出格式？
3. 它需要参考哪些 Schema、规则或示例？
4. 遇到不确定情况应该如何处理？
5. 当前是否已有一版可用提示词，还是需要从零起草？

示例答案：

```text
角色：你是信息抽取助手。
要求：只输出 JSON，不输出解释。
字段：entities、relations、uncertain_cases。
不确定时：保留 evidence_span，并把 confidence 设为 low。
当前版本：没有，需要先生成 baseline prompt。
```

### 4. STATUS.md：当前状态、版本、阻塞项、下一步

需要问：

1. 当前工作焦点是什么？本轮优先围绕哪 1 到 3 条主线推进？
2. 当前思考层级是什么？业务、数据、执行、评估，还是项目管理？
3. 当前项目处于哪个阶段？刚启动、已有 baseline、正在修数据、正在跑实验？
4. 当前已知版本是什么？不知道可以写“未建立”。
5. 当前最大阻塞是什么？下一步最应该推进哪个盒子？

示例答案：

```text
当前工作焦点：先完成 baseline，再补齐验收指标。
当前思考层级：项目启动规划。
阶段：刚初始化，还没有 baseline。
当前版本：task-v1、schema-v1，数据和 skill 尚未建立。
阻塞：缺少 20 条代表性样例和验收指标。
下一步：先进入业务与指标盒，补 TASK_SPEC.md。
```

## 推荐提问格式

```markdown
我发现当前项目还缺少以下初始化信息：

| 文件 | 缺少什么 | 为什么需要 |
| --- | --- | --- |
| TASK_SPEC.md | 输入样例、输出格式、主指标 | 决定任务边界和验收方式 |
| data/schema.yaml | 标签集和边界规则 | 决定 GT、Skill 和评估是否对齐 |

请先回答这 5 个问题：
1. 项目要解决的业务问题是什么？
2. 给 1 条典型输入样例。
3. 期望输出长什么样？
4. 第一阶段最关心的指标是什么？
5. 哪些情况先不处理？

示例回答：
目标：……
输入样例：……
输出示例：……
主指标：……
非目标：……
```

## 可以暂时占位的写法

当用户还不知道答案时，不要卡死流程。可以写：

```text
暂定：待业务确认。
风险：该字段会影响后续评估可比性。
下次确认：进入业务与指标盒时优先补齐。
```
