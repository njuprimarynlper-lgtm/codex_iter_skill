# 编辑授权边界与可修改文件白名单协议

目标：防止 AI 擅自修改未确认过的文件。所有项目应在根目录维护 `EDIT_SCOPE.md`，作为“哪些文件能改、哪些必须先确认、哪些禁止改”的单一真相源。

## 触发条件

- 准备创建、修改、移动或删除任何文件。
- 用户问“哪些文件能改”“为什么改了这个文件”“以后别动某些文件”。
- 项目没有 `EDIT_SCOPE.md`，但本轮需要写文件。
- 任务会影响 GT、指标、验收线、生产配置、历史结论、实验报告或长期记忆。

## 基本规则

1. 写文件前先读 `EDIT_SCOPE.md`；如果不存在，按“无持久授权”处理。
2. 无持久授权时，只能修改用户本轮明确点名的文件；其他拟修改文件必须先确认。
3. 白名单只降低“是否需要确认文件范围”的成本，不覆盖风险分级。GT、指标、验收线、生产配置、历史结论和不可逆操作仍按高风险确认。
4. `EDIT_SCOPE.md` 自身的持久变更必须向用户说明，因为它会影响后续授权边界。
5. 匹配不到任何规则的文件默认 `confirm_required`。

## 推荐文件结构

```markdown
# 编辑授权边界

## 匹配顺序
1. forbidden
2. confirm_required
3. always_allowed
4. generated_outputs
5. task_scoped
6. unmatched -> confirm_required

## always_allowed
| 模式 | 用途 | 备注 |
| --- | --- | --- |
| `STATUS.md` | 当前状态和交接 | 仅限短状态，不写长期历史 |
| `PROJECT_MEMORY.md` | 当前有效长期记忆 | 改变当前最优/历史归档前仍需确认 |
| `docs/README.md` | 文档索引 | 不写长正文 |
| `docs/archive/**` | 文档归档 | 保留来源和归档原因 |
| `tmp/**` | 临时诊断产物 | 不作为正式结论 |

## generated_outputs
| 模式 | 用途 |
| --- | --- |
| `experiments/**/report.md` | 单次实验人工报告 |
| `experiments/**/report.json` | 单次实验结构化报告 |

## confirm_required
| 模式 | 原因 |
| --- | --- |
| `data/**` | 数据、GT 或样本可能影响评估 |
| `eval_kit/**` | 指标和评估口径 |
| `pipeline/**` | 生产流程 |
| `autokg_pipeline/**` | 业务/模型逻辑 |
| `tests/**` | 验证口径会影响结论可信度 |
| `experiments/CHANGELOG.md` | 实验历史和决策记录 |

## forbidden
| 模式 | 原因 |
| --- | --- |
| `.git/**` | Git 内部数据 |
| `.env` | 本地密钥 |
| `**/*secret*` | 潜在密钥 |

## task_scoped
| 时间 | 用户授权 | 模式 | 过期条件 |
| --- | --- | --- | --- |
```

## 执行步骤

1. 收集本轮拟写文件清单，包含新增、修改、移动、删除。
2. 读取 `EDIT_SCOPE.md` 并按匹配顺序分类：
   - `allowed`：可直接修改，但仍受风险分级约束。
   - `task_scoped`：仅本轮可改；不要自动变成长期授权。
   - `confirm_required`：必须说明原因和影响，等用户确认。
   - `forbidden`：拒绝修改，除非用户明确改变授权边界。
   - `unmatched`：按 `confirm_required`。
3. 若白名单缺失，先输出建议的初始白名单草案；本轮只修改用户明确点名或已确认的文件。
4. 用户确认后再执行写入，并在最终回复列出实际修改文件和授权来源。
5. 如果本轮发现反复需要改同类文件，建议把对应模式加入 `EDIT_SCOPE.md`，但持久加入前必须确认。
