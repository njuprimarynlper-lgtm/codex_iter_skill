# 用户目的识别与命令对齐协议

目标：模型执行用户命令前，先判断用户为什么要做这件事，避免机械执行字面动作而偏离真实目标。这个协议不要求长篇解释；它要求在行动前完成一次短促、明确的目标对齐。

## 触发条件

- 用户给出任何需要执行的命令、修改、分析、提交、上传、运行测试或生成产物的请求。
- 用户的命令是手段型表达，例如“更新一下”“跑一下”“分析一下”“上传”“删掉”“改成”“同步”。
- 当前任务可能影响文件、指标、GT、实验结论、生产配置、项目焦点或远端仓库。
- 字面命令看起来能执行，但可能不是达成用户目的的最佳路径。

## 执行前判断

执行前先在心里完成四个判断；必要时用一句话告诉用户：

1. 用户真正想解决什么问题？
2. 用户希望得到什么产物或状态变化？
3. 当前字面命令是否直接服务这个目的？
4. 是否存在风险更低、证据更充分或更贴近目的的执行方式？

## 直接执行

满足以下条件时可以直接执行，并在过程中简短说明判断：

- 用户目的清晰，字面命令就是合理手段。
- 风险低或已在当前授权范围内。
- 不会改变 GT、指标、验收线、生产配置、历史结论或远端状态。
- 不会修改未确认过的文件，或已通过 `EDIT_SCOPE.md` / 本轮明确授权。

示例表达：

```text
我理解你的目的不是只改一行规则，而是让后续 agent 执行命令前先对齐目标。我会把它写成触发规则和检查协议。
```

## 先确认

满足任一条件时，先说明目的判断和冲突点，再等用户确认：

- 字面命令与推测目的不一致。
- 有多个合理目的，会导致不同执行路径。
- 字面命令可能造成高风险影响。
- 存在更安全或更直接的替代方案，但会偏离用户原话。
- 需要改变 GT 标准、指标口径、验收线、生产配置、历史结论、项目约束或远端发布状态。

确认格式：

```markdown
我理解你的目的可能是：……
字面命令会导致：……
更稳妥的做法是：……
是否按这个方向执行？
```

## 不能做的事

- 不要把“思考目的”变成拖延执行；目的清晰时应继续推进。
- 不要用模型臆测替代用户明确命令；只能在有冲突或风险时请求确认。
- 不要忽略项目文件里的当前焦点、项目约束、编辑白名单和历史结论。
- 不要为了完成字面命令而破坏用户真实目标，例如用户要“上传最新 skill”，却只同步本地不提交远端。

## 交接记录

当目的判断影响了执行路径时，在最终回复或 `STATUS.md` 交接中说明：

- 识别到的用户目的。
- 因目的对齐而调整的执行方式。
- 用户确认过的例外或后续约束。
