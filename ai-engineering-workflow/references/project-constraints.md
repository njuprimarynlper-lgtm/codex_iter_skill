# 项目独有约束协议

目标：为每个项目提供一个稳定入口，记录该项目特有的硬约束、默认禁止项、默认假设和冲突处理规则。模型进入项目和执行任务前必须检查这些约束，避免把通用 skill 行为套到不适用的项目上。

## 文件位置

项目根目录维护 `PROJECT_CONSTRAINTS.md`。它记录“这个项目有什么特别要求”，不是任务目标、当前状态或文件编辑白名单。

| 文件 | 负责内容 |
| --- | --- |
| `PROJECT_CONSTRAINTS.md` | 项目独有约束、默认禁止项、默认假设、冲突处理 |
| `TASK_SPEC.md` | 业务目标、输入输出、指标和验收线 |
| `STATUS.md` | 当前工作焦点、当前阶段、阻塞、下一步和交接 |
| `PROJECT_MEMORY.md` | 当前有效结论、当前最优、废弃结论和下一轮优先级 |
| `EDIT_SCOPE.md` | 哪些文件可改、哪些要确认、哪些禁止改 |

## 触发条件

- 进入项目、新窗口恢复、开工检查、下一步诊断或会话收尾。
- 执行任何可能受项目规则影响的任务，例如改 prompt、改 workflow、改评估、改 GT、分析 badcase、生成报告、切换模型或服务。
- 用户说“这个项目必须”“这个项目不要”“以后都按”“这里不能”“只允许”等稳定约束。
- 当前指令看起来和已有项目约束冲突。

## 必须遵守

1. 进入项目时先读 `PROJECT_CONSTRAINTS.md`；如果不存在，说明项目暂无独立约束文件，不要编造约束。
2. 项目约束优先于通用默认行为；但不能覆盖系统/开发者安全规则。
3. 当前用户指令与项目约束冲突时，先说明当前约束、冲突点和影响，等待用户确认临时豁免或更新约束。
4. 持久修改 `PROJECT_CONSTRAINTS.md` 属于高风险项目边界变更，必须说明新旧约束差异并得到确认。
5. 如果约束涉及 GT、标注规则、验收线、评估口径或生产配置，必须同时走对应确认协议和影响链检查。
6. 只适用于当前项目的偏好写入 `PROJECT_CONSTRAINTS.md`；跨项目反复出现的行为 gap 才更新 skill。
7. 不要在约束文件里保存密钥、token、内部地址明文或不可公开凭据；只保存引用位置和使用规则。

## 执行步骤

1. 读取 `PROJECT_CONSTRAINTS.md`，提取与当前任务相关的约束。
2. 对照当前用户请求，判断是否存在冲突或高风险边界变化。
3. 若无冲突，按约束执行，并在必要时把使用到的约束写入交接。
4. 若有冲突，输出：

| 当前约束 | 本轮请求 | 冲突影响 | 需要用户确认 |
| --- | --- | --- | --- |

5. 用户确认临时豁免时，只在本轮执行，并在 `STATUS.md` 交接中记录。
6. 用户确认长期变更时，更新 `PROJECT_CONSTRAINTS.md` 的正文和版本历史；涉及 GT、指标、生产配置或历史结论时，同步影响链检查。
