# AI 工程化工作流协议

这些协议承接主 `SKILL.md` 中不适合长期堆叠的细则。使用时只读取与当前任务相关的协议，不需要一次性读取全文。

## 1. 下一步诊断协议

触发条件：

- 用户问“下一步做什么”“今天做什么”“继续”“帮我看当前状态”“我们现在该推进什么”。
- `STATUS.md` 缺少当前阶段、阻塞项或下一步。
- 项目状态不清楚，但用户希望继续推进。

执行步骤：

1. 读取 `STATUS.md` 和 `TASK_SPEC.md`；如果存在 `PROJECT_INTAKE.md`，也读取它。
2. 判断当前阶段：刚初始化、信息不足、需要建立 baseline、需要补数据、需要跑实验、需要汇总、需要决策或需要收尾。
3. 输出未来 3 到 5 步路线图，包含推荐顺序、所属盒子、预期产出和依赖。
4. 给出首选方案和备选方案，说明适用条件、代价和风险。
5. 如果信息足够，先执行推荐顺序中的第一步；如果信息不足，一次最多问 5 个会阻塞行动的问题。
6. 用户回答后，把信息写入对应项目文件。

推荐输出结构：

```markdown
## 当前判断
- 当前思考层级：
- 当前项目阶段：
- 主要阻塞：

## 推荐路线图
| 顺序 | 建议动作 | 所属层级/盒子 | 为什么现在做 | 预期产出 |
| --- | --- | --- | --- | --- |

## 候选方案
- 首选方案：
- 备选方案 A：
- 备选方案 B：

## 立即执行

## 需要你补充的信息

## 我会更新的文件
```

## 2. 不明确指令确认协议

当用户指令不明确时，先判断歧义是否会影响：

- 方向选择
- 文件修改
- 数据或 GT
- 版本或生产配置
- 实验结论
- 不可逆操作

如果会影响，必须先确认，不要擅自执行。确认时输出：

1. 当前理解。
2. 歧义点。
3. 推荐默认方案。
4. 1 到 3 个最关键问题。

不要把能从项目文件中读取的信息再问用户。

如果只是低风险实现细节，且不会改变用户目标或重要产物，可以声明“我先按某假设处理”并继续执行；相关假设要在交接或 `STATUS.md` 中标为待确认。

## 3. GT 标准变更确认协议

每次涉及以下内容时，必须先确认 GT 标准：

- GT 数据修改。
- 标注规则修改。
- 验收标准修改。
- 边界定义修改。
- 由 GT 派生的评估结论更新。

确认前必须向用户说明：

1. 当前 GT 标准是什么。
2. 拟变更的标准是什么。
3. 受影响的数据、指标和实验结论是什么。
4. 是否会导致旧实验不可直接比较。

只有用户确认标准无误后，才能更新 GT 标准文件、GT 数据或由 GT 派生的评估结论。

## 4. 方案迭代测试门

每次执行方案迭代后，包括修改 Skill、Workflow、数据处理、评估脚本或关键配置，都必须进入测试门。

执行步骤：

1. 说明本轮方案迭代改了什么。
2. 询问用户是否跑一遍测试或评估。
3. 如果已有明确测试命令且用户已授权继续，可以直接运行。
4. 如果没有授权，说明建议测试范围、预计产出和是否需要用户确认。
5. 测试完成后，对照 `TASK_SPEC.md` 中的验收线和护栏指标判断是否达标。

如果达标，输出：

- 达标依据。
- 关键指标。
- 护栏指标。
- 是否建议晋升或进入下一轮探索。

如果不达标，必须输出：

- 当轮指标。
- 目标阈值。
- 差距。
- 主要问题。
- 典型 badcase。
- 可能原因。
- 下一轮候选方案。

如果无法测试，说明原因、缺少什么、风险是什么，并把“未验证”写入交接记录或 `STATUS.md`。

## 5. 项目文档归档协议

给人看的输出文件必须统一由当前项目管理。

默认位置：

- 使用手册、设计文档、阶段总结、评审记录、决策说明等写入项目 `docs/` 或项目已有文档目录。
- 单次实验的人工报告 `report.md` 留在对应 `experiments/` 目录。

禁止做法：

- 不要把项目专属交付文档写进 skill 目录。
- 不要写到全局目录或零散临时目录。
- 不要只把重要结论留在对话里。

skill 目录只保存可复用的规则、模板、脚本和 reference。若发现项目专属的人类可读文档被写到 skill 目录，应迁移回项目文档目录，并在回复中给出项目内路径。

写给人的使用手册时，重点写“人需要做什么、AI 会帮什么、最后得到什么、如何验收”。内部配置字段、登记格式或机器可读契约应放到设计文档或 reference，而不是作为手册主线。

## 6. 实验比较与晋升协议

实验比较前确认：

- 指标定义一致，或指标被显式作为实验变量。
- 数据切分和 GT 版本一致，或数据被显式作为实验变量。
- Schema、model、temperature、seed 和 runtime 假设被固定，或被显式作为实验变量。
- failed、invalid、missing run 被单独说明，不混入正常均值比较。

实验汇总应包括：

- 按主指标排序的 Top 实验。
- 可比较前提下的因素主效应。
- 交互效应或条件性行为。
- 指标、GT、Schema 兼容性警告。
- 合并、回滚或下一轮 grid 建议。

晋升前确认：

- 主指标优于 baseline。
- 护栏指标没有超出容忍范围。
- 关键 badcase 没有业务不可接受回归。
- 被晋升的 artifact 已复制或引用到稳定的 `pipeline/` 路径。
- `STATUS.md` 和 `experiments/CHANGELOG.md` 记录晋升原因。

不要静默修改已完成的实验目录。实验有误时，标记为 invalid，并重新生成新实验。

## 7. 中文文件编码协议

在 Windows/PowerShell 中生成中文 Markdown、CSV 或报告时，避免把包含中文字符串常量的 here-string 通过管道传给 `python -` 或其他解释器；PowerShell 可能按当前代码页转换源码，导致中文标题被永久写成 `?`。

优先做法：

- 使用已有 UTF-8 脚本文件。
- 使用 `apply_patch` 创建或修改 UTF-8 文件。
- 让临时代码只含 ASCII 字符，并用 `\uXXXX` 转义生成中文。
- 写出文件时显式使用 UTF-8。
- 交付前用 `rg -n "\?{3,}|\x{FFFD}" <输出.md>` 自检；命中时必须重新生成或修复。
