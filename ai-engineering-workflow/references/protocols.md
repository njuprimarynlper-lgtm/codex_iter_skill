# AI 工程化工作流协议

这些协议承接主 `SKILL.md` 中不适合长期堆叠的细则。使用时只读取与当前任务相关的协议，不需要一次性读取全文。

## 1. 下一步诊断协议

触发条件：

- 用户问“下一步做什么”“今天做什么”“继续”“帮我看当前状态”“我们现在该推进什么”。
- `STATUS.md` 缺少当前阶段、阻塞项或下一步。
- 项目状态不清楚，但用户希望继续推进。

执行步骤：

1. 读取 `STATUS.md` 和 `TASK_SPEC.md`；如果存在 `PROJECT_INTAKE.md`，也读取它。
2. 判断当前阶段：刚初始化、信息不足、需要建立 baseline、需要补数据、需要跑实验、需要汇总、需要决策或需要收尾。
3. 输出未来 3 到 5 步路线图，包含推荐顺序、所属盒子、预期产出和依赖。
4. 给出首选方案和备选方案，说明适用条件、代价和风险。
5. 如果信息足够，先执行推荐顺序中的第一步；如果信息不足，一次最多问 5 个会阻塞行动的问题。
6. 用户回答后，把信息写入对应项目文件。

推荐输出结构：

```markdown
## 当前判断
- 当前思考层级：
- 当前项目阶段：
- 主要阻塞：

## 推荐路线图
| 顺序 | 建议动作 | 所属层级/盒子 | 为什么现在做 | 预期产出 |
| --- | --- | --- | --- | --- |

## 候选方案
- 首选方案：
- 备选方案 A：
- 备选方案 B：

## 立即执行

## 需要你补充的信息

## 我会更新的文件
```

## 2. 不明确指令确认协议

当用户指令不明确时，先判断歧义是否会影响：

- 方向选择
- 文件修改
- 数据或 GT
- 版本或生产配置
- 实验结论
- 不可逆操作

如果会影响，必须先确认，不要擅自执行。确认时输出：

1. 当前理解。
2. 歧义点。
3. 推荐默认方案。
4. 1 到 3 个最关键问题。

不要把能从项目文件中读取的信息再问用户。

如果只是低风险实现细节，且不会改变用户目标或重要产物，可以声明“我先按某假设处理”并继续执行；相关假设要在交接或 `STATUS.md` 中标为待确认。

## 3. 新日修改前 Git 提交检查协议

触发条件：

- 当前日期已经晚于上次工作日期，且即将进行文件修改、逻辑修改、实验配置修改或项目文档修改。
- 用户说“新的一天”“今天继续”“继续昨天的工作”等，且项目可能已有历史改动。
- 进入项目后发现存在未提交 Git 改动，且这些改动可能来自上一轮工作。

执行步骤：

1. 在真正修改文件前，检查当前目录是否是 Git 仓库。
2. 如果是 Git 仓库，查看 `git status --short`，判断是否存在未提交改动。
3. 若存在未提交改动，先向用户说明当前有历史改动，并询问是否需要先提交 Git。
4. 询问时给出建议，例如“建议先提交上一轮稳定改动，再开始今天的新修改”或“这些像是当前任务内改动，可以继续”。
5. 未经用户明确确认，不要自动 `git add`、`git commit` 或 `git push`。
6. 如果用户选择暂不提交，继续工作前要在回复或 `STATUS.md` 中标记“存在未提交历史改动，用户选择暂不提交”。
7. 如果当前不是 Git 仓库，说明无法做提交检查，然后继续当前任务。

推荐询问格式：

```markdown
我看到这是新的一天，且当前仓库存在未提交改动。

当前判断：
- 历史改动范围：
- 是否像上一轮遗留：
- 建议：

是否需要我先帮你提交上一轮改动到 Git，再开始今天的新修改？
```

## 4. GT 标准变更确认协议

每次涉及以下内容时，必须先确认 GT 标准：

- GT 数据修改。
- 标注规则修改。
- 验收标准修改。
- 边界定义修改。
- 由 GT 派生的评估结论更新。

确认前必须向用户说明：

1. 当前 GT 标准是什么。
2. 拟变更的标准是什么。
3. 受影响的数据、指标和实验结论是什么。
4. 是否会导致旧实验不可直接比较。

只有用户确认标准无误后，才能更新 GT 标准文件、GT 数据或由 GT 派生的评估结论。

## 5. 影响链检查与修正协议

触发条件：

- 业务标准、GT、评测基准、验收线、边界规则或评估口径发生变化。
- Skill、Prompt、Workflow、模型调用流程、数据处理、评估脚本、报告格式或生产配置发生变化。
- 完成一次文件或逻辑修改后，判断该修改可能影响其他盒子的输入、输出、指标、实验结论或说明文档。
- 用户说“标准变了”“这会影响哪些地方”“帮我同步一下”“自动检查一下是否漏改”等。

核心原则：

- 自动检查不等于静默修改。模型必须先自动识别影响链并主动提醒用户哪些地方可能需要修改。
- 涉及业务标准、GT、评测基准、验收线、生产配置、历史实验结论或高风险数据修改时，必须先向用户确认是否修改、修改哪些项。
- 对于用户已明确授权的低风险说明文档同步，可以直接更新，但最终回复必须列出已更新文件。
- 如果用户暂不确认修正，必须把待修正项写入 `STATUS.md`、`experiments/CHANGELOG.md` 或本轮交接摘要中的待办。

执行步骤：

1. 识别触发源：本轮发生了什么变化，归属哪个盒子，是否涉及标准、数据、执行、评估或项目管理。
2. 读取最小必要文件：
   - 总览：`STATUS.md`、`TASK_SPEC.md`。
   - 数据/GT：`data/README.md`、`data/schema.yaml`、相关数据版本说明。
   - 执行方案：`pipeline/skill_prompt.txt`、`pipeline/config.yaml`、`pipeline/workflow_config.yaml` 或相关变体文件。
   - 评估迭代：`experiments/CHANGELOG.md`、相关 `report.md`/`report.json`。
   - 人类文档：`docs/README.md` 及本次受影响的使用手册、设计文档或决策说明。
3. 输出影响链清单，至少包含：
   - 受影响层级/盒子。
   - 可能需要修改的文件或产物。
   - 为什么受影响。
   - 建议动作。
   - 风险等级。
   - 是否需要用户确认。
4. 主动给出推荐修正顺序：先修标准和任务定义，再修数据/执行方案，再修评估和文档；如具体项目有更强依赖，以项目依赖为准。
5. 向用户确认是否执行修正：
   - 若影响项少，直接列出文件并问是否按建议修改。
   - 若影响项多，给出“首选修正包”和“可暂缓项”，请用户选择。
6. 用户确认后，只修改被确认的项；修改完成后再次执行一次影响链复查，确认没有新增遗漏。
7. 如果本轮修正涉及方案迭代，继续进入“方案迭代测试门”。

推荐输出结构：

```markdown
## 影响链检查
- 触发源：
- 当前归属盒子：
- 是否涉及高风险标准/评估/生产配置：

| 层级/盒子 | 可能受影响的文件或产物 | 为什么受影响 | 建议动作 | 风险 | 是否需要你确认 |
| --- | --- | --- | --- | --- | --- |

## 推荐修正顺序
1.
2.
3.

## 需要你确认
是否按上述建议修改这些项？如果只想先改一部分，请指出要修改的文件或层级。

## 暂不修改时的记录
我会把未修正项写入：
```

## 6. 方案迭代测试门

每次执行方案迭代后，包括修改 Skill、Workflow、数据处理、评估脚本或关键配置，都必须进入测试门。

执行步骤：

1. 说明本轮方案迭代改了什么。
2. 询问用户是否跑一遍测试或评估。
3. 如果已有明确测试命令且用户已授权继续，可以直接运行。
4. 如果没有授权，说明建议测试范围、预计产出和是否需要用户确认。
5. 测试完成后，对照 `TASK_SPEC.md` 中的验收线和护栏指标判断是否达标。

如果达标，输出：

- 达标依据。
- 关键指标。
- 护栏指标。
- 是否建议晋升或进入下一轮探索。

如果不达标，必须输出：

- 当轮指标。
- 目标阈值。
- 差距。
- 主要问题。
- 典型 badcase。
- 可能原因。
- 下一轮候选方案。

如果无法测试，说明原因、缺少什么、风险是什么，并把“未验证”写入交接记录或 `STATUS.md`。

## 7. 逻辑修改文档同步协议

每次修改以下逻辑后，必须同步更新相应说明文档：

- Skill 主规则、协议、模板或脚本逻辑。
- Workflow、Prompt、Skill 变体或模型调用流程。
- 数据处理、GT 标准、标注规则或边界定义。
- 评估指标、测试逻辑、实验矩阵或报告格式。
- 项目流程、协作协议、文档归档规则或晋升/回滚规则。

执行步骤：

1. 判断本次修改属于哪个盒子，以及影响哪些人类可读说明文档。
2. 先查看主 `SKILL.md` 的“说明文档索引”和项目 `docs/README.md`。
3. 更新对应说明文档；如果没有合适文档，创建或更新 `docs/README.md` 记录新文档位置和用途。
4. 若修改影响人如何操作，更新使用手册。
5. 若修改影响系统接口、协议、验证门、模板或脚本，更新设计文档。
6. 若修改影响任务目标、数据、指标或实验结论，更新对应的 `TASK_SPEC.md`、`data/README.md`、`experiments/CHANGELOG.md` 或实验报告。
7. 最终回复必须列出“逻辑改动”和“已同步的说明文档位置”。

如果暂时无法更新说明文档，不能假装完成；必须在回复和 `STATUS.md` 或交接记录中写明“文档未同步”的原因和后续补齐动作。

## 8. 项目文档归档协议

给人看的输出文件必须统一由当前项目管理。

默认位置：

- 使用手册、设计文档、阶段总结、评审记录、决策说明等写入项目 `docs/` 或项目已有文档目录。
- 单次实验的人工报告 `report.md` 留在对应 `experiments/` 目录。

禁止做法：

- 不要把项目专属交付文档写进 skill 目录。
- 不要写到全局目录或零散临时目录。
- 不要只把重要结论留在对话里。

skill 目录只保存可复用的规则、模板、脚本和 reference。若发现项目专属的人类可读文档被写到 skill 目录，应迁移回项目文档目录，并在回复中给出项目内路径。

写给人的使用手册时，重点写“人需要做什么、AI 会帮什么、最后得到什么、如何验收”。内部配置字段、登记格式或机器可读契约应放到设计文档或 reference，而不是作为手册主线。

## 9. 实验比较与晋升协议

实验比较前确认：

- 指标定义一致，或指标被显式作为实验变量。
- 数据切分和 GT 版本一致，或数据被显式作为实验变量。
- Schema、model、temperature、seed 和 runtime 假设被固定，或被显式作为实验变量。
- failed、invalid、missing run 被单独说明，不混入正常均值比较。

实验汇总应包括：

- 按主指标排序的 Top 实验。
- 可比较前提下的因素主效应。
- 交互效应或条件性行为。
- 指标、GT、Schema 兼容性警告。
- 合并、回滚或下一轮 grid 建议。

晋升前确认：

- 主指标优于 baseline。
- 护栏指标没有超出容忍范围。
- 关键 badcase 没有业务不可接受回归。
- 被晋升的 artifact 已复制或引用到稳定的 `pipeline/` 路径。
- `STATUS.md` 和 `experiments/CHANGELOG.md` 记录晋升原因。

不要静默修改已完成的实验目录。实验有误时，标记为 invalid，并重新生成新实验。

## 10. 中文文件编码协议

在 Windows/PowerShell 中生成中文 Markdown、CSV 或报告时，避免把包含中文字符串常量的 here-string 通过管道传给 `python -` 或其他解释器；PowerShell 可能按当前代码页转换源码，导致中文标题被永久写成 `?`。

优先做法：

- 使用已有 UTF-8 脚本文件。
- 使用 `apply_patch` 创建或修改 UTF-8 文件。
- 让临时代码只含 ASCII 字符，并用 `\uXXXX` 转义生成中文。
- 写出文件时显式使用 UTF-8。
- 交付前用 `rg -n "\?{3,}|\x{FFFD}" <输出.md>` 自检；命中时必须重新生成或修复。
