# AI 算法研发管家型 Skill 协议

这些协议承接主 `SKILL.md` 中不适合长期堆叠的细则。使用时只读取与当前任务相关的协议，不需要一次性读取全文。协议的目标是让模型像算法研发管家一样主动完成开工检查、层级路由、缺口识别、影响链检查、测试复盘和交接维护。

## 0. 当前工作焦点动态更新协议

触发条件：

- 新窗口、新会话或新一天恢复项目。
- 用户问“下一步做什么”“继续”“当前状态”“今天做什么”。
- 用户明确提出新主线、优先级变化或阶段变化。
- 本轮任务完成、测试完成、不达标、受阻或出现新的高优先级风险。
- 模型准备给出的建议与 `STATUS.md` 的当前工作焦点不一致。

核心原则：

- 当前工作焦点必须写入 `STATUS.md`，不能只留在当前对话窗口。
- 下一步建议必须围绕当前工作焦点展开；若建议偏离焦点，必须说明偏离原因。
- 用户给出明确局部执行任务时，例如分析单条新数据、单个样本、单个 badcase、一个文件片段或一条 GT/预测差异，必须先执行该任务；当前工作焦点只能作为背景，不得把局部任务改写成稳定性、整体指标、实验汇总或路线图分析。
- 用户明确说“现在先做 A”“本轮聚焦 A 和 B”时，可以直接更新焦点并在最终回复说明。
- 模型仅凭推断要替换、降级、暂停、归档当前焦点时，必须先问用户确认。
- 同一项目可以有 1 到 3 条并行主线，但必须给出推荐顺序，避免“什么都重要”。

`STATUS.md` 推荐字段：

```markdown
## 当前工作焦点
- 主线：
- 推荐顺序：
- 当前阶段：
- 正在处理的问题：
- 本轮不做：
- 最近焦点变化：
- 焦点来源：
- 待用户确认：
```

执行步骤：

1. 读取 `PROJECT_CONSTRAINTS.md`（如果存在）、`STATUS.md`、`PROJECT_MEMORY.md`（如果存在）和 `TASK_SPEC.md`。
2. 检查 `STATUS.md` 是否包含“当前工作焦点”；若缺失，先从最近交接、项目记忆和用户当前请求中提取候选焦点。
3. 判断用户当前请求与焦点关系：
   - 明确局部执行任务：直接完成用户指定动作，最多补充它与当前焦点的关系，不自动切换任务。
   - 直接相关：围绕焦点推进。
   - 部分相关：说明哪些主线相关，哪些可能偏离。
   - 不相关：先提醒“这与当前焦点不一致”，给出继续当前焦点或切换焦点的选择。
4. 判断是否需要更新焦点：
   - 明确更新：用户直接指定主线、优先级、阶段或本轮不做事项。
   - 建议更新：测试结果、阻塞或影响链表明焦点应调整，但需要用户确认。
   - 不更新：只是当前焦点内的一个执行动作。
5. 更新 `STATUS.md` 的“当前工作焦点”和“交接”；必要时同步 `PROJECT_MEMORY.md` 的“下一轮优先级”。
6. 输出下一步时先展示焦点，再给路线图、候选方案和首选建议。

推荐输出结构：

```markdown
## 当前工作焦点
- 主线：
- 推荐顺序：
- 当前阶段：
- 本轮不做：

## 焦点判断
- 当前请求与焦点关系：
- 是否需要更新焦点：
- 原因：

## 围绕焦点的推荐路线
| 顺序 | 动作 | 对应主线 | 为什么现在做 | 预期产出 |
| --- | --- | --- | --- | --- |

## 需要你确认
```

## 1. 下一步诊断协议

触发条件：

- 用户问“下一步做什么”“今天做什么”“继续”“帮我看当前状态”“我们现在该推进什么”。
- `STATUS.md` 缺少当前工作焦点、当前阶段、阻塞项或下一步。
- 项目状态不清楚，但用户希望继续推进。

执行步骤：

1. 读取 `PROJECT_CONSTRAINTS.md`（如果存在）、`STATUS.md`、`PROJECT_MEMORY.md`（如果存在）和 `TASK_SPEC.md`；如果存在 `PROJECT_INTAKE.md`，也读取它。
2. 先执行“当前工作焦点动态更新协议”：识别焦点、判断当前请求是否相关、必要时更新或请求确认。
3. 判断当前阶段：刚初始化、信息不足、需要建立 baseline、需要补数据、需要跑实验、需要汇总、需要决策或需要收尾。
4. 输出未来 3 到 5 步路线图，必须围绕当前工作焦点展开，包含推荐顺序、所属盒子、预期产出和依赖。
5. 给出首选方案和备选方案，说明适用条件、代价和风险；如果存在与焦点无关但可做的事项，放入“不建议本轮做”。
6. 如果信息足够，先执行推荐顺序中的第一步；如果信息不足，一次最多问 5 个会阻塞行动的问题。
7. 用户回答后，把信息写入对应项目文件，必要时同步更新当前工作焦点。

推荐输出结构：

```markdown
## 当前判断
- 当前工作焦点：
- 当前思考层级：
- 当前项目阶段：
- 主要阻塞：

## 推荐路线图
| 顺序 | 建议动作 | 所属层级/盒子 | 为什么现在做 | 预期产出 |
| --- | --- | --- | --- | --- |

## 候选方案
- 首选方案：
- 备选方案 A：
- 备选方案 B：

## 不建议本轮做

## 立即执行

## 需要你补充的信息

## 我会更新的文件
```

## 1.1 开工管家检查协议

触发条件：

- 用户说“开工”“今天开始”“今天继续”“继续昨天的工作”“我们今天从哪开始”。
- 新会话进入已有项目，用户希望恢复工作状态。
- 用户给出轻量指令，但项目状态、未验证项、待确认项或历史改动可能影响下一步；不包括已经给出具体样本、样本 ID、文件片段、命令或代码位置的明确执行指令。

执行步骤：

1. 读取 `PROJECT_CONSTRAINTS.md`（如果存在）、`STATUS.md`、`PROJECT_MEMORY.md`（如果存在）、`TASK_SPEC.md` 和 `docs/README.md`；如果存在 `experiments/CHANGELOG.md`，也读取最近一轮记录。
2. 判断当前是否是新的一天或新会话恢复；如果是新的一天且后续可能写入文件，包括固化昨日结论/方法，先进入“新日修改前 Git 提交检查协议”。
3. Git 检查完成、用户选择已记录或确认当前不是 Git 仓库后，再进入“新日有效结论与方法固化协议”。
4. 执行“当前工作焦点动态更新协议”，确认新窗口或新一天应优先围绕哪些主线推进。
5. 检查 `STATUS.md` 是否包含当前工作焦点、当前阶段、当前层级、阻塞项、待确认事项和下一步；缺失时进入“下一步诊断协议”或 `references/intake-guide.md`。
6. 检查是否存在未验证方案、未完成实验、缺失报告、待确认标准、文档未同步或影响链待修正项；发现问题时进入“主动巡检协议”。
7. 输出今日推荐工作路线，包含 3 到 5 步、推荐顺序、首选方案、候选方案和需要用户确认的问题。
8. 如果第一步明确且不涉及高风险确认，直接执行第一步；如果涉及标准、GT、生产配置、历史结论或不可逆操作，先向用户确认。

推荐输出结构：

```markdown
## 开工检查
- 当前工作焦点：
- 当前项目阶段：
- 当前推荐进入层级：
- 昨日待固化结论/方法：
- 昨日/上一轮遗留：
- 未验证或待确认事项：

## 今日推荐路线
| 顺序 | 动作 | 所属层级/盒子 | 依赖 | 预期产出 |
| --- | --- | --- | --- | --- |

## 首选方案与候选
- 首选方案：
- 候选方案 A：
- 候选方案 B：

## 需要你确认
1.

## 我将先执行
```

## 1.2 新日有效结论与方法固化协议

触发条件：

- 新的一天开始工作，用户说“今天继续”“继续昨天”“新的一天开工”等。
- `STATUS.md`、上一轮交接、`PROJECT_MEMORY.md` 或 `experiments/CHANGELOG.md` 显示前一天形成了结论、方法、约定、候选最佳方案或新的工作习惯。
- 前一天完成了测试、复盘、影响链检查、方案晋升、文档同步或一轮探索收尾。
- `STATUS.md` 中存在“昨日待固化”“待确认结论”“待沉淀方法”等字段。

核心原则：

- 固化是把“昨天形成的有效结论和方法”写入长期可复用位置，不是把所有历史搬进 `STATUS.md`。
- `STATUS.md` 只保留今天开工需要的当前状态和指向；固化后的结论进入 `PROJECT_MEMORY.md`、`experiments/CHANGELOG.md`、项目文档或稳定配置。
- 固化通常会写入 `PROJECT_MEMORY.md`、`experiments/CHANGELOG.md`、项目文档或 `STATUS.md`，因此新一天执行实质固化写入前，必须先完成新日 Git 提交检查或记录当前不是 Git 仓库。
- 不确定是否该固化的内容必须先问用户，不能替用户判断成长期结论。
- 凡是会影响指标、GT、验收、生产配置、稳定 Skill/Workflow、当前最优结果、历史归档或实验结论含义的固化，必须先确认。
- 明确是低风险的文档格式、交接摘要整理、已确认结论搬运，可以直接固化并在最终回复说明。

固化位置：

| 内容类型 | 固化位置 |
| --- | --- |
| 当前有效判断、当前最优结果、下一轮优先级 | `PROJECT_MEMORY.md` |
| 实验决策、方案晋升/放弃、不可比较说明 | `experiments/CHANGELOG.md` |
| 单次实验指标、badcase、输入 token、输出 token、thinking token 占比、失败率 | 对应 `experiments/*/report.md` / `report.json` |
| 人如何使用的新方法、流程或注意事项 | 项目 `docs/` 中的使用手册、设计文档或决策说明 |
| 可复用到所有项目的 Skill 协议或模板 | Skill 的 `references/` 或脚本；需要用户明确确认 |
| 今天需要继续执行的短期状态 | `STATUS.md` |

不确定必须询问的典型情况：

- 结论只来自一次现象，还没有测试或复盘支撑。
- 结论依赖特定 GT、指标、样本集、模型版本或业务口径。
- 前一天有多个候选方案，无法判断哪个是当前最优。
- 方法可能只是当前项目临时做法，不确定是否应变成通用 Skill 规则。
- 固化会覆盖旧摘要、改变当前最优结果、归档历史结论或影响用户后续执行方式。
- 结论和当前工作焦点冲突，或会改变今天的推荐路线。

执行步骤：

1. 确认“新日修改前 Git 提交检查协议”已完成、用户选择已记录或当前不是 Git 仓库；否则先返回 Git 检查，不进行固化写入。
2. 读取 `STATUS.md`、`PROJECT_MEMORY.md`（如果存在）、上一轮交接摘要；必要时读取 `experiments/CHANGELOG.md` 和最近报告。
3. 提取前一天候选固化项，分为：
   - 有效结论。
   - 有效方法。
   - 当前最优结果。
   - 待确认假设。
   - 应归档历史。
   - 仅作临时记录、不建议固化。
4. 为每一项判断证据等级：
   - 已确认：用户明确确认、测试达标、复盘结论清楚。
   - 建议确认：有证据但会影响方向、结论、历史或用户行为。
   - 不建议固化：证据不足、只是一时现象、已被覆盖或不可比较。
5. 输出固化草案；对“建议确认”的项目逐条问用户是否固化、固化到哪里。
6. 用户确认后执行写入：
   - 更新 `PROJECT_MEMORY.md` 的当前有效结论、当前最优结果、下一轮优先级或已固化方法。
   - 更新 `STATUS.md` 的当前工作焦点、当前阶段、下一步、最近固化日期和待确认项。
   - 需要时更新 `experiments/CHANGELOG.md` 或人类说明文档。
7. 如果用户暂不确认，把待固化项写入 `STATUS.md` 的“新日固化”或“待用户确认”，不要静默丢弃。

推荐输出结构：

```markdown
## 新日固化检查
| 候选项 | 类型 | 证据 | 建议动作 | 固化位置 | 是否需要你确认 |
| --- | --- | --- | --- | --- | --- |

## 我建议直接固化
- 

## 需要你确认
1. 

## 暂不固化
- 
```

## 1.3 操作授权与确认协议

触发条件：

- 用户要求修改文件、数据、配置、实验结论、文档或项目状态。
- 模型准备主动执行某个动作，但不确定是否可以直接做。
- 指令比较轻量，例如“顺手改一下”“继续”“帮我处理”，需要判断是否会影响关键产物。

核心原则：

- 先分级，再执行。
- 低风险事项可以直接做，最终回复说明改了什么。
- 中风险事项先说明影响和默认方案；如果用户已经给出明确授权，可以继续执行。
- 高风险事项必须先问用户，未经确认不得修改。
- 无法判断风险等级时，按高风险处理。

### A. 可直接做

满足以下条件时，AI 可以直接执行，事后汇报：

- 文档格式、错别字、标点、标题层级、链接修复。
- 模板补充、说明文档同步、文档索引登记。
- 非关键说明文字润色，不改变业务含义。
- 新增草稿、候选文档、示例片段或低风险辅助说明。
- 不影响生产配置、指标口径、GT、实验结论和历史归档的轻量整理。

执行要求：

1. 可以直接改。
2. 最终回复列出文件和改动摘要。
3. 如果发现改动实际影响关键产物，立即停止并升级为“必须确认”。

### B. 先提醒再做

满足以下条件时，AI 应先说明影响、默认方案和可能风险；如果用户已经授权当前任务，可以继续：

- 新增普通 Skill、Prompt 或 Workflow 候选变体，但不替换稳定版本。
- 调整实验报告格式、字段说明或汇总展示方式。
- 归档历史摘要、合并重复说明，但不改变当前最优结果和历史结论含义。
- 修改非生产配置、示例配置、草稿模板。
- 改动可能影响后续维护方式，但不直接影响业务验收或生产运行。

执行要求：

1. 简短说明“我准备这样处理”。
2. 标明是否会影响稳定版本、实验可比性或用户使用方式。
3. 如用户已明确授权本轮处理，可以继续；否则询问一句确认。

### C. 必须确认

凡是涉及以下事项，必须先向用户确认，不能静默执行：

- 指标、主指标、护栏指标、评估口径或验收线。
- GT 标准、标注规则、边界定义、数据版本或数据删除/覆盖。
- 生产配置、稳定 Skill、稳定 Workflow、默认模型或运行参数。
- 实验结论、历史结论、当前最优结果、方案晋升或回滚。
- 长期记忆中的历史归档、不可比较标记、摘要合并后覆盖旧摘要。
- Git 提交、推送、删除、覆盖、批量移动等不可逆或难回滚操作。
- 用户明确要求“先问我”“不要直接改”的任何范围。

确认时必须说明：

1. 当前理解。
2. 准备修改什么。
3. 为什么属于高风险。
4. 推荐方案。
5. 需要用户确认的问题。

推荐输出结构：

```markdown
## 操作授权判断
- 动作：
- 风险等级：可直接做 / 先提醒再做 / 必须确认
- 原因：
- 推荐处理：

## 需要你确认
```

## 2. 不明确指令确认协议

当用户指令不明确时，先判断歧义是否会影响：

- 方向选择
- 文件修改
- 数据或 GT
- 版本或生产配置
- 实验结论
- 不可逆操作

如果会影响，必须先确认，不要擅自执行。确认时输出：

1. 当前理解。
2. 歧义点。
3. 推荐默认方案。
4. 1 到 3 个最关键问题。

不要把能从项目文件中读取的信息再问用户。

如果只是低风险实现细节，且不会改变用户目标或重要产物，可以声明“我先按某假设处理”并继续执行；相关假设要在交接或 `STATUS.md` 中标为待确认。

## 3. 新日修改前 Git 提交检查协议

触发条件：

- 当前日期已经晚于上次工作日期，且即将进行文件修改、逻辑修改、实验配置修改或项目文档修改。
- 用户说“新的一天”“今天继续”“继续昨天的工作”等，且项目可能已有历史改动。
- 进入项目后发现存在未提交 Git 改动，且这些改动可能来自上一轮工作。

执行步骤：

1. 在真正修改文件前，检查当前目录是否是 Git 仓库。
2. 如果是 Git 仓库，查看 `git status --short`，判断是否存在未提交改动。
3. 若存在未提交改动，先向用户说明当前有历史改动，并询问是否需要先提交 Git。
4. 询问时给出建议，例如“建议先提交上一轮稳定改动，再开始今天的新修改”或“这些像是当前任务内改动，可以继续”。
5. 未经用户明确确认，不要自动 `git add`、`git commit` 或 `git push`。
6. 如果用户选择暂不提交，继续工作前要在回复或 `STATUS.md` 中标记“存在未提交历史改动，用户选择暂不提交”。
7. 如果当前不是 Git 仓库，说明无法做提交检查，然后继续当前任务。

推荐询问格式：

```markdown
我看到这是新的一天，且当前仓库存在未提交改动。

当前判断：
- 历史改动范围：
- 是否像上一轮遗留：
- 建议：

是否需要我先帮你提交上一轮改动到 Git，再开始今天的新修改？
```

## 4. GT 标准变更确认协议

每次涉及以下内容时，必须先确认 GT 标准：

- GT 数据修改。
- 标注规则修改。
- 验收标准修改。
- 边界定义修改。
- 由 GT 派生的评估结论更新。
- 任何准备依据 GT 文件中的 `confidence` 判定硬/软 GT、筛选评估样本、刷新 GT 或形成指标结论的操作。

确认前必须向用户说明：

1. 当前 GT 标准是什么。
2. 拟变更的标准是什么。
3. 受影响的数据、指标和实验结论是什么。
4. 是否会导致旧实验不可直接比较。

只有用户确认标准无误后，才能更新 GT 标准文件、GT 数据或由 GT 派生的评估结论。

GT 文件中的 `confidence` 只能作为候选排序、复核优先级和历史不确定性记录，不得自动等同于 GT 标准、硬/软边界、评估阈值或验收口径。使用任何 confidence 阈值前，必须把阈值含义、受影响样本和指标影响列给用户核实；未核实前只能标为待裁定或实验视角，不进入正式 GT、评估结论或当前最优方案比较。

如果拟刷新或新增的 GT 规则、标注边界、评估口径与已有标准或用户历史裁定冲突，必须暂停修改并让用户重新判断。不得静默覆盖旧规则、把两套标准混合使用，或为了让某次预测/指标更好而自行选择一边。冲突项应先记录为待裁定，不进入正式 GT、评估结论、当前最优方案比较或对外演示材料。

## 5. 影响链检查与修正协议

触发条件：

- 业务标准、GT、评测基准、验收线、边界规则或评估口径发生变化。
- Skill、Prompt、Workflow、模型调用流程、数据处理、评估脚本、报告格式或生产配置发生变化。
- 完成一次文件或逻辑修改后，判断该修改可能影响其他盒子的输入、输出、指标、实验结论或说明文档。
- 用户说“标准变了”“这会影响哪些地方”“帮我同步一下”“自动检查一下是否漏改”等。

核心原则：

- 自动检查不等于静默修改。模型必须先自动识别影响链并主动提醒用户哪些地方可能需要修改。
- 涉及业务标准、GT、评测基准、验收线、生产配置、历史实验结论或高风险数据修改时，必须先向用户确认是否修改、修改哪些项。
- 对于用户已明确授权的低风险说明文档同步，可以直接更新，但最终回复必须列出已更新文件。
- 如果用户暂不确认修正，必须把待修正项写入 `STATUS.md`、`experiments/CHANGELOG.md` 或本轮交接摘要中的待办。
- 若触发源是模型/API 访问服务切换，且项目要求服务只能硬编码切换，必须把被指定的源码配置位置视为唯一真相源；影响链检查要主动查找并处理可能覆盖它的环境变量、service profile、launcher 参数、外层 `PipelineConfig` 注入和文档默认值。

执行步骤：

1. 识别触发源：本轮发生了什么变化，归属哪个盒子，是否涉及标准、数据、执行、评估或项目管理。
2. 读取最小必要文件：
   - 总览：`STATUS.md`、`TASK_SPEC.md`。
   - 数据/GT：`data/README.md`、`data/schema.yaml`、相关数据版本说明。
   - 执行方案：`pipeline/skill_prompt.txt`、`pipeline/config.yaml`、`pipeline/workflow_config.yaml` 或相关变体文件。
   - 评估迭代：`experiments/CHANGELOG.md`、相关 `report.md`/`report.json`。

3. 输出影响链清单，至少包含：

4. 主动给出推荐修正顺序：先修标准和任务定义，再修数据/执行方案，再修评估和文档；如具体项目有更强依赖，以项目依赖为准。

7. 如果本轮修正涉及方案迭代，继续进入“方案迭代测试门”。

## 5.1 主动巡检协议

触发条件：

- 用户说“帮我检查一下当前项目”“看下有没有遗漏”“当前 skill 是否还一致”“继续前先扫一遍”。
- 开工管家检查发现项目状态不清、待办堆积、实验报告缺失、文档未同步、未验证改动或待确认标准。
- 完成一轮较大修改后，需要确认是否还有漏改、漏测或漏记。

巡检范围：

- 状态巡检：`STATUS.md` 是否包含当前工作焦点、当前阶段、当前层级、阻塞项、下一步和待确认事项。
- 记忆巡检：`PROJECT_MEMORY.md` 是否存在，是否只保留当前有效结论和当前最优结果，历史摘要是否已归档；若项目已有多轮实验、GT/指标裁定、当前最优或废弃结论而该文件不存在，必须进入“长期记忆压缩与结果归档协议”，输出创建草案，不能只提醒缺失。
- 标准巡检：`TASK_SPEC.md`、数据说明、评估口径和验收线是否存在不一致或待确认。
- 执行巡检：Skill、Prompt、Workflow、配置或生产入口是否存在未验证改动。
- 实验巡检：实验是否缺少 `report.json`、`report.md`、badcase、输入 token、输出 token、thinking token 占输出 token 比例、失败率或达标判断。
- 文档巡检：逻辑改动后，使用手册、设计文档、`docs/README.md` 或交接记录是否同步。

执行步骤：

1. 先说明本次巡检范围，不要无限扩大到所有文件；默认只读最小必要文件。
2. 输出问题清单，区分“必须先处理”“建议本轮处理”“可暂缓观察”。
3. 对每个问题说明影响、建议动作、是否需要用户确认和推荐顺序。
4. 涉及高风险标准、GT、评估结论、生产配置或历史实验可比性时，先确认再修改。
5. 用户确认后，只修正被确认的项；修正后再次快速巡检，确认没有新增遗漏。

推荐输出结构：

```markdown
## 主动巡检结果
| 优先级 | 问题 | 影响 | 建议动作 | 是否需要你确认 |
| --- | --- | --- | --- | --- |

## 推荐处理顺序
1.
2.
3.

## 我建议本轮先处理

## 可暂缓项

## 需要你确认
```

### 主动巡检提醒规则

主动巡检不能只依赖用户知道这个功能。满足以下任一条件时，即使用户没有主动要求巡检，也要在回复中提醒：

- 新用户或首次进入一个项目，`STATUS.md` 中没有最近巡检记录。
- 开工管家检查发现上一轮遗留、未验证方案、待确认标准、缺失报告或文档未同步。
- 用户问“下一步做什么”时，项目已经推进过一轮以上，但状态、实验、文档或测试记录不完整。
- 完成一次方案迭代、影响链修正或文档同步后，仍可能存在漏测、漏记或漏改。
- 会话收尾时存在未验证改动、待确认事项或下次启动依赖。

提醒方式：

1. 不要只说“可以巡检”。要说明为什么此时建议巡检。
2. 如果风险较高，直接进入主动巡检协议或询问用户是否现在执行。
3. 如果只是低风险提示，把它放在回复末尾的“巡检提醒”中，不打断当前主任务。
4. 给用户一条可直接复制的指令，例如：`先帮我巡检一下当前项目有没有漏改、漏测、漏记。`
5. 如果执行了巡检或用户选择暂不巡检，要把结果或选择写入 `STATUS.md` 的交接或巡检状态。

推荐提醒格式：

```markdown
## 巡检提醒
我注意到当前项目存在：未验证改动/待确认标准/实验报告缺失/文档可能未同步。
建议本轮或收尾前做一次主动巡检，检查是否有漏改、漏测、漏记。
你可以直接说：先帮我巡检一下当前项目有没有漏改、漏测、漏记。
```

## 6. 方案迭代测试门

每次执行方案迭代后，包括修改 Skill、Workflow、数据处理、评估脚本或关键配置，都必须进入测试门。

执行步骤：

1. 说明本轮方案迭代改了什么。
2. 从 `PROJECT_MEMORY.md`、`STATUS.md`、`experiments/CHANGELOG.md` 或最近 `report.json`/`report.md` 中找出上一轮或当前最优结果的数据，作为本轮对照基线。
3. 如果找不到上一轮最好数据，必须说明缺失位置和风险，不要凭印象编造；必要时先建议补齐历史报告或把本轮标记为“无可比基线”。
4. 判断是否是该项目第一次测试或评估：检查 `STATUS.md` 的测试记录和展示偏好、`TASK_SPEC.md` 的“测试结果展示偏好”、`experiments/CHANGELOG.md` 以及是否已有 `experiments/*/report.json`/`report.md`。
5. 如果是首次测试且尚未确认展示偏好，先询问用户“测试完成后希望展示哪些信息”，候选项包括：主指标、护栏指标、输入 token、输出 token、thinking token、thinking token 占输出 token 比例、测试用例 ID、失败用例、典型 badcase、日志路径、报告路径、模型/环境版本；确认后写入 `TASK_SPEC.md` 或 `STATUS.md`。除非用户明确要求，测试报告默认不展示等待时间/耗时。
6. 询问用户是否跑一遍测试或评估。
7. 如果已有明确测试命令且用户已授权继续，可以直接运行。
8. 如果没有授权，说明建议测试范围、预计产出和是否需要用户确认。
9. 测试完成后，先按已确认的展示偏好输出结果，再对照 `TASK_SPEC.md` 中的验收线、护栏指标和上一轮最好数据判断是否达标、是否进步、是否可晋升。

如果达标，输出：

- 达标依据。
- 关键指标。
- 护栏指标。
- 用户确认的展示字段。
- 与上一轮或当前最优结果的指标差异。
- 比较口径是否一致。
- 是否建议晋升或进入下一轮探索。

如果不达标，必须输出：

- 当轮指标。
- 上一轮或当前最优结果指标。
- 用户确认的展示字段。
- 目标阈值。
- 差距。
- 主要问题。
- 典型 badcase。
- 可能原因。
- 下一轮候选方案。

如果无法测试，说明原因、缺少什么、风险是什么，并把“未验证”写入交接记录或 `STATUS.md`。

## 7. 逻辑修改文档同步协议

每次修改以下逻辑后，必须同步更新相应说明文档：

- Skill 主规则、协议、模板或脚本逻辑。
- Workflow、Prompt、Skill 变体或模型调用流程。
- 数据处理、GT 标准、标注规则或边界定义。
- 评估指标、测试逻辑、实验矩阵或报告格式。
- 项目流程、协作协议、文档归档规则或晋升/回滚规则。

执行步骤：

1. 判断本次修改属于哪个盒子，以及影响哪些人类可读说明文档。
2. 先查看主 `SKILL.md` 的“说明文档索引”和项目 `docs/README.md`。
3. 更新对应说明文档；如果没有合适文档，创建或更新 `docs/README.md` 记录新文档位置和用途。
4. 若修改影响人如何操作，更新使用手册。
5. 若修改影响系统接口、协议、验证门、模板或脚本，更新设计文档。
6. 若修改影响任务目标、数据、指标或实验结论，更新对应的 `TASK_SPEC.md`、`data/README.md`、`experiments/CHANGELOG.md` 或实验报告。
7. 最终回复必须列出“逻辑改动”和“已同步的说明文档位置”。

如果暂时无法更新说明文档，不能假装完成；必须在回复和 `STATUS.md` 或交接记录中写明“文档未同步”的原因和后续补齐动作。

## 8. 行为纠偏与 Skill 自动更新协议

触发条件：

- 用户用同一种方式纠正 AI 行为两次及以上，例如反复要求补字段、先确认、不要展示某类信息、必须提交/打包/对比/记录。
- 同类偏好、边界或工作方式在多个任务中反复出现，说明 Skill 默认行为和用户期望存在稳定 gap。
- 用户明确说“以后都这样”“把这个写进 skill”“你又忘了同一件事”。

执行步骤：

1. 明确列出 gap：
   - AI 默认行为是什么。
   - 用户期望行为是什么。
   - 已重复出现的证据或最近触发点是什么。
2. 判断归属：
   - 可复用到 AI 研发工作流：更新 Skill 主规则、相关协议、模板或脚本。
   - 只适用于当前项目：更新 `STATUS.md`、`TASK_SPEC.md`、`PROJECT_MEMORY.md` 或项目文档。
   - 涉及 GT、指标、验收线、生产配置或历史结论：先进入对应确认协议，不得绕过用户确认。
3. 对可复用规则，不要只口头承诺；直接修改 Skill，并同步 `D:\skill` 打包目录和压缩包。
4. 修改后执行最小验证，例如规则检索、脚本编译、压缩包缓存检查。
5. 如果用户已要求提交，使用中文 commit 描述提交并推送；否则在最终回复说明“已更新但未提交”。

## 9. 项目文档归档协议

给人看的输出文件必须统一由当前项目管理。

默认位置：

- 使用手册、设计文档、阶段总结、评审记录、决策说明等写入项目 `docs/` 或项目已有文档目录。
- 单次实验的人工报告 `report.md` 留在对应 `experiments/` 目录。

禁止做法：

- 不要把项目专属交付文档写进 skill 目录。
- 不要写到全局目录或零散临时目录。
- 不要只把重要结论留在对话里。

skill 目录只保存可复用的规则、模板、脚本和 reference。若发现项目专属的人类可读文档被写到 skill 目录，应迁移回项目文档目录，并在回复中给出项目内路径。

写给人的使用手册时，重点写“人需要做什么、AI 会帮什么、最后得到什么、如何验收”。内部配置字段、登记格式或机器可读契约应放到设计文档或 reference，而不是作为手册主线。

## 10. 实验比较与晋升协议

触发条件：

- 用户要求汇总、比较、晋升、回滚、分析实验结果或处理无效实验。
- 用户在新窗口说“使用这个 Skill，在这个工程中，分析最新的测试结果”“分析最新一轮结果”“看看最新测试怎么样”等轻量指令。
- 方案迭代后已有测试输出，需要和上一轮或当前最优结果比较。

最新测试结果分析要求：

1. 先读取 `STATUS.md` 的当前工作焦点、`PROJECT_MEMORY.md` 的当前最优结果和下一轮优先级，以及 `experiments/CHANGELOG.md` 最近记录。
2. 自动定位最新测试结果：优先使用 `STATUS.md`、`PROJECT_MEMORY.md` 或 `CHANGELOG` 明确标注的最新实验；如果没有标注，再按 `experiments/*/report.json` / `report.md` 的最近更新时间、目录命名或 comparison group 推断。
3. 如果存在多个“最新”候选且无法判断，列出候选、选择依据和风险，向用户确认；不要要求用户重新补全部上下文。
4. 读取最新报告和上一轮或当前最优基线，输出本轮指标、护栏指标、是否达标、与基线差异、可比性判断、主要问题、典型 badcase、可能原因和下一步 3 到 5 步建议。
5. 如果缺少报告、指标口径不一致或 GT/Schema/model 不可比，明确说明缺失位置和风险，把本轮标记为“无法直接比较”或“需补齐报告”，不要凭印象判断。
6. 如果最新结果形成阶段性结论，继续进入“长期记忆压缩与结果归档协议”；如果只是待确认信号，写入 `STATUS.md` 的待确认事项。

实验比较前确认：

- 指标定义一致，或指标被显式作为实验变量。
- 数据切分和 GT 版本一致，或数据被显式作为实验变量。
- Schema、model、temperature、seed 和 runtime 假设被固定，或被显式作为实验变量。
- failed、invalid、missing run 被单独说明，不混入正常均值比较。
- 每次提出下一轮迭代方案或实验 grid 前，必须列出上一轮或当前最优结果的数据：方案 ID、报告路径、主指标、护栏指标、数据/GT/Schema/model 版本和适用条件。

实验汇总应包括：

- 按主指标排序的 Top 实验。
- 上一轮或当前最优结果与本轮 Top 实验的 delta。
- 可比较前提下的因素主效应。
- 交互效应或条件性行为。
- 指标、GT、Schema 兼容性警告。
- 合并、回滚或下一轮 grid 建议。

晋升前确认：

- 主指标优于 baseline。
- 在同一比较口径下优于上一轮或当前最优结果。
- 护栏指标没有超出容忍范围。
- 关键 badcase 没有业务不可接受回归。
- 被晋升的 artifact 已复制或引用到稳定的 `pipeline/` 路径。
- `STATUS.md` 和 `experiments/CHANGELOG.md` 记录晋升原因。

不要静默修改已完成的实验目录。实验有误时，标记为 invalid，并重新生成新实验。

## 11. 长期记忆压缩与结果归档协议

触发条件：

- 形成阶段性结论，例如一轮实验汇总完成、某个方案被确认胜出、某个方向被放弃。
- `STATUS.md`、`experiments/CHANGELOG.md`、实验报告或对话摘要开始变长，影响开工读取效率。
- `PROJECT_MEMORY.md` 不存在，但项目已经存在多轮实验、GT/指标裁定、当前最优候选、废弃结论、不可比较历史，或 `STATUS.md` 已超过 Markdown 生命周期预算。
- 准备晋升最佳方案、收尾一轮探索、开始新一轮探索或新一天开工前需要快速恢复状态。
- 用户说“压缩一下记忆”“只保留当前有效结论”“把历史结果归档”“合并这些摘要”等。

核心原则：

- `PROJECT_MEMORY.md` 只放当前仍有决策价值的压缩记忆，不复制完整实验历史。
- 阶段性结论形成后，只保留一个当前最优结果作为推荐方案；历史候选、旧摘要和已被覆盖的结论进入归档区或 `experiments/CHANGELOG.md`。
- “只保留最优结果”不是删除历史文件，而是把历史从当前工作记忆中移出，保留可追溯链接和归档原因。
- 多个阶段摘要可以合并；合并后必须保留来源、时间、适用条件、覆盖关系和是否仍有效。
- 如果指标、GT、评估口径或验收线变化导致旧结果不可比较，必须把旧结果标记为不可直接比较，不得继续作为当前最优。

执行步骤：

1. 读取最小必要文件：`STATUS.md`、`PROJECT_MEMORY.md`（如果存在）、`experiments/CHANGELOG.md`、本轮相关 `report.md`/`report.json` 和必要的 `TASK_SPEC.md`。
2. 判断本轮是否形成阶段性结论：是否有明确最佳方案、是否有被放弃方向、是否有待确认假设、是否有不可比较历史。
3. 如果 `PROJECT_MEMORY.md` 不存在且上述条件成立，输出“创建 PROJECT_MEMORY.md 草案”；如果已存在，则输出“压缩/更新草案”。草案分为：
   - 当前有效结论。
   - 当前最优结果。
   - 已归档历史结果。
   - 待确认假设。
   - 不可比较或已废弃结论。
   - 下一轮优先级。
4. 如果会改变“当前最优结果”或归档历史结论，先向用户确认；低风险摘要整理可以直接执行，但最终必须列出更新文件。
5. 用户确认后，更新 `PROJECT_MEMORY.md`：
   - 当前推荐只保留一个最优结果。
   - 次优、失败、旧口径或已覆盖方案进入归档区。
   - 合并重复摘要，保留来源列表和更新时间。
6. 同步更新 `STATUS.md` 的当前阶段、当前最优、下一步和最近记忆压缩时间。
7. 若涉及实验结论归档或晋升，同步更新 `experiments/CHANGELOG.md`，说明归档原因、保留的最优结果和不可比较风险。

## 12. 中文文件编码协议

在 Windows/PowerShell 中生成中文 Markdown、CSV 或报告时，避免把包含中文字符串常量的 here-string 通过管道传给 `python -` 或其他解释器；PowerShell 可能按当前代码页转换源码，导致中文标题被永久写成 `?`。

优先做法：

- 使用已有 UTF-8 脚本文件。
- 使用 `apply_patch` 创建或修改 UTF-8 文件。
- 让临时代码只含 ASCII 字符，并用 `\uXXXX` 转义生成中文。
- 写出文件时显式使用 UTF-8。
- 交付前用 `rg -n "\?{3,}|\x{FFFD}" <输出.md>` 自检；命中时必须重新生成或修复。
